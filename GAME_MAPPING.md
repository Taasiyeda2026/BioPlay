# מיפוי מלא – BioPlay

> מסמך מיפוי בלבד (ללא תיקוני קוד), עבור ניווט, מבנה, נתונים, תלותים, אי-התאמות והצעת ארגון להמשך.

## 1) מיפוי כל המסכים והעמודים במשחק

| מזהה/עמוד | סוג | קובץ מקור | איך מגיעים | מה מוצג למשתמש | פעולות עיקריות |
|---|---|---|---|---|---|
| `index.html` | דף כניסה | `index.html` | פתיחת האתר | כרטיס ברוכים הבאים + 3 כפתורי מסלול | מעבר ל-`biohebrew.html`, `bioarab.html`, `bioplay.html` |
| `biohebrew.html` | פעילות/מצגת חטיבה עברית | `biohebrew.html` | כפתור “חטיבה (עברית)” | Hero + 7 שלבים (`data-step=0..6`) + ניווט קדימה/אחורה + טיימר | `startLesson`, `goToStep`, חשיפת תשובות/טיפים |
| `bioarab.html` | פעילות/מצגת חטיבה ערבית | `bioarab.html` | כפתור “חטיבה (ערבית)” | מבנה מקביל לעברית בערבית | ניווט שלבים ותצוגות תוכן |
| `bioplay.html` | משחק דלתות ראשי | `bioplay.html` + `doors.json` | כפתור “פעילות ליסודי” | מסך בית, מסך חדר דלתות, מסך סיום, ומסכים דינמיים לכל דלת/שלב | פתיחת משחק, בחירת דלת, מענה שאלות, הזנת קודים, פתיחת מנעול, סיכום |

### מסכים פנימיים ב-`bioplay.html` (דינמיים)

| מזהה מסך | נוצר ע"י | תיאור |
|---|---|---|
| `screen-home` | `renderHome()` | מסך פתיחה + סטטוס נעילה/פתיחה + הגדרת מספר דלתות |
| `screen-doors` | `renderDoors()` | גריד דלתות אקראי (לפי `doorsCount`) + מצב הושלם/נעול |
| `door-{id}-intro` | `renderDoorIntro(id)` | הקדמת דלת עם תמונת אורגניזם וטקסט intro |
| `door-{id}-q-{step}` | `renderStep(id, step)` | שאלה רב-ברירה (שלבים 1..3) |
| `door-{id}-code-{step}` | `renderCodeScreen(id, step)` | הזנת קוד מעבר עבור השלב |
| `door-{id}-confirm-{step}` | `renderConfirmation(id, step)` | אישור תשובה נכונה + מעבר לשלב הבא/מנעול סופי |
| `door-{id}-final-lock` | `renderFinalLock(id)` | הזנת קוד פתיחת דלת סופי |
| `door-{id}-summary` | `renderDoorSummary(id)` | סיכום דלת (אורגניזם/יכולת/השראה) |
| `screen-final` | `renderFinal()` | מסך ברכות כללי לאחר השלמת כל הדלתות שנבחרו |

## 2) מיפוי מבנה הקבצים ותלויות

| קובץ | תפקיד | תלוי ב | בשימוש ע"י |
|---|---|---|---|
| `index.html` | שער כניסה וניווט למסלולים | Google Fonts | משתמש קצה (כניסה ראשונית) |
| `bioplay.html` | מנוע משחק (UI + לוגיקה JS inline) | `doors.json`, `images/*`, Google Apps Script API | משתמש קצה במסלול יסודי |
| `doors.json` | מאגר תוכן קבוע של הדלתות (1..30) | קבצי `images/*` לפי `image`, לוגיקת `bioplay.html` | `bioplay.html` בלבד |
| `biohebrew.html` | פעילות חטיבה בעברית (מצגת אינטראקטיבית) | תמונות שונות + JS inline | משתמש קצה במסלול חטיבה עברית |
| `bioarab.html` | פעילות חטיבה בערבית | תמונות שונות + JS inline | משתמש קצה במסלול חטיבה ערבית |
| `images/*` | נכסי מדיה (דלתות, חיות, אייקונים, וידאו) | — | כל דפי המערכת לפי צורך |

## 5) מיפוי הנתונים הקבועים במשחק

- מקור דאטה מרכזי: `doors.json` (30 דלתות × 3 שלבים).
- שדות קבועים לכל דלת: `id`, `title`, `creature`, `image`, `intro`, `steps[]`, `summary`, `finalLock`, `finalLockCode`.
- שדות קבועים לכל שלב: `title`, `question`, `options[]`, `answer`, `code`, הודעות UI (כישלון/הצלחה), טקסטי כפתורים.
- טקסטי מערכת קבועים בקוד (`bioplay.html`): הודעות סטטוס, כותרות מסכים, fallback לטעינת קבצים/קוד שגוי, טקסטי כפתורי ברירת מחדל.
- מזהים קבועים: `GAME_ID="main_game"`, URL ל-Apps Script, שמות מסכי DOM (`screen-home`, `screen-doors`, `screen-final`, ועוד דינמיים).

### כפילויות / חיבורים קשים לתחזוקה

- רוב לוגיקת המשחק נמצאת בקובץ HTML יחיד (`bioplay.html`) עם JS inline גדול – מקשה על בדיקות, reuse ותחזוקה.
- טקסטים רבים משוכפלים (`confirmText`, `codeWrongText`, `transitionText`) בכל דלת/שלב במקום תבניות מרכזיות.
- קיימת תלות כפולה לקוד הסופי (`finalLock.code` וגם `finalLockCode`) שעלולה להתפצל בעתיד.
- מספר הדלת למשתמש (`displayNumber`) אקראי בכל סבב, בעוד השיוך הפנימי הוא `id`; זה מקשה דיבוג ותיעוד פדגוגי.

## 8) הצעה לארגון טוב יותר להמשך (ללא ביצוע שינוי בפועל)

1. **פיצול שכבות**: להעביר JS מ-`bioplay.html` ל-`src/game/*.js` (state, renderer, navigation, services).
2. **סכמת תוכן קשיחה**: להגדיר `schema` ל-`doors.json` (למשל JSON Schema) ולאמת ב-CI.
3. **אחידות מזהים**: לבחור מזהה אחד לדלת (או `id` או `displayNumber`) ולהציגו בעקביות.
4. **קטלוג טקסטים קבועים**: להוציא הודעות מערכת/כפתורים ל-`i18n` או constants file.
5. **Asset manifest**: קובץ מרכזי שמאמת שכל `image` בנתונים קיים באמת.
6. **טבלת ניווט רשמית**: לשמור מסמך state-machine קצר שמתעד כל מעבר ומצבי שגיאה.
7. **בדיקות אוטומטיות לתוכן**: בדיקה קבועה ל-`answer in options`, title/id match, uniqueness, image existence.

## 9) סיכום קצר

- המערכת בנויה סביב דף משחק אחד (`bioplay.html`) + מאגר נתונים (`doors.json`) + שער כניסה (`index.html`).
- זוהו פערי תוכן נקודתיים (תשובות, כותרות סיכום, ותמונות חסרות) שמצדיקים תיקון לפני הרחבה עתידית.
- יש בסיס טוב למשחק, אך מומלץ ליישם ארגון מודולרי וסכמת נתונים כדי להקל על שינויים מדויקים בהמשך.
